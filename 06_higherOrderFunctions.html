Partial Application

(a,b) vs. a => b =>

// multiple arguments
const times = (a,b) => a x b;
times(2);// returns undefined or something else...not number

// argument chain, curried
const times = a => b => a * b;
times(2), // returns a function

Partial Application:Map


const times = a => b => a * b
const twoTimes = times(2)

// the 3 versions are kind of all the same (communication is a bit different)
[1,2,3].map(x => times(2)(x));
[1,2,3].map(times(2)); // eta reduction
[1,2,3].map(x => twoTimes); // other reference, point-free-notation
Outer-type and cardinality(size) stays fix when mapping. Inner-type can change when mapping.

Filter

x => x % 2 === 1 // fucntion needs to return a predicate (boolean)
--> filter out what doenst fit the check (retain all in java)
outer type still array, cardinality can change, inner type stays

Example:
const odd = x => x % 2 === 1;
[1, 2, 3].filter(x => x % 2 === 1);
[1, 2, 3].filter(x => odd(x));
// eta reduction [1, 2, 3].filter(odd);

Reduce

reduce((acc, cur) => acc + cur) // accumulator & commulator
Does not keep the outer-type necessarily, f.e. from array to a number

in alphabetically order --> in other languages then JS don't rename ! use acc & cur

Example: "un-partial" reduce const plus = (accu, cur) => accu + cur;
[1, 2, 3].reduce((accu, cur) => accu + cur);
[1, 2, 3].reduce(plus);
// variant with initial accu value as 2nd argument
// then cur starts at first element
[1, 2, 3].reduce(plus, 0);
// option to set start element: 0 is the startelement, to prevent problems f.e when the array is empty.
